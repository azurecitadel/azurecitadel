## Introduction

The previous page covered encryption at rest — protecting data that is stored. Azure Confidential Computing (ACC) addresses the next boundary: **encryption in use**, protecting data while it is actively being processed.

Without confidential computing, data must be decrypted in memory before a workload can operate on it. That decrypted data is theoretically accessible to a privileged user, the hypervisor, or even Microsoft's own operators. ACC closes that gap by running workloads inside a **Trusted Execution Environment (TEE)** — a hardware-enforced isolated region where memory is encrypted by the CPU itself and inaccessible to anything outside, including the host OS, the hypervisor, and cloud operators.

{{< flash >}}
The three complementary layers of data protection:

- **At rest** — data encrypted on disk, keys held in Key Vault (covered in the previous page).
- **In transit** — data encrypted over the wire with TLS.
- **In use** — data encrypted in memory during processing, enforced by TEE hardware.

CMK and ACC are complementary. SKR, covered on the next page, is the mechanism that bridges them — releasing a key only to a verified TEE.
{{< /flash >}}

## Confidential VMs

### AMD SEV-SNP

Confidential VMs run on AMD processors with Secure Encrypted Virtualization – Secure Nested Paging (SEV-SNP). The hardware encrypts the VM's entire memory using a key generated by the CPU that the hypervisor and host OS never see.

This is the most accessible ACC option — it requires no application code changes. Any existing workload can be lifted into a Confidential VM and gain hardware memory encryption immediately.

VM sizes in the [DCasv5 / ECasv5 series](https://learn.microsoft.com/azure/virtual-machines/dcasv5-dcadsv5-series) support AMD SEV-SNP. The guest OS sees the attestation report via the vTPM, and Azure provides tooling to exchange that report for an Azure Attestation token.

### Intel SGX

Intel Software Guard Extensions (SGX) provide application-level isolated regions — enclaves — that are protected by the CPU. Unlike Confidential VMs, SGX enclaves are not whole-VM: only the code and data explicitly placed inside the enclave boundary are protected.

SGX requires the application to be written (or ported) using the Open Enclave SDK, Intel SGX SDK, or a compatible framework such as [Gramine](https://gramineproject.io/) (which can run unmodified applications). This makes SGX more complex to adopt but gives very fine-grained control over what is isolated.

VM sizes in the [DCsv2 / DCsv3 series](https://learn.microsoft.com/azure/virtual-machines/dcv3-series) support Intel SGX.

## Confidential Containers

### Azure Container Instances (ACI)

Azure Container Instances supports a confidential computing SKU that runs container groups inside an AMD SEV-SNP TEE. The attestation is at the container group boundary rather than per-container.

Microsoft provides an open-source [confidential sidecar](https://github.com/Azure/confidential-sidecars) container that handles attestation and Secure Key Release on behalf of the workload container. Your application calls a local REST endpoint exposed by the sidecar — no custom attestation code needed.

This makes confidential ACI a practical starting point for containerised workloads that need SKR without the overhead of managing a full VM.

#### References

- [Confidential containers on Azure Container Instances](https://learn.microsoft.com/azure/container-instances/container-instances-confidential-overview)
- [Confidential sidecars (GitHub)](https://github.com/Azure/confidential-sidecars)
- [SKR flow for confidential containers in ACI](https://learn.microsoft.com/azure/confidential-computing/skr-flow-confidential-containers-azure-container-instance)

### Azure Kubernetes Service (AKS)

AKS supports confidential node pools using the same AMD SEV-SNP hardware as Confidential VMs. Each node in a confidential node pool runs in a hardware-isolated TEE, and workloads inherit the memory encryption without code changes.

AKS is a good fit for containerised workloads that need confidential computing at scale, particularly when combined with workload identity for SKR.

#### References

- [Confidential VMs in AKS node pools](https://learn.microsoft.com/azure/aks/use-cvm)
- [Workload identity with AKS](https://learn.microsoft.com/azure/aks/workload-identity-overview)

### Azure Container Apps (ACA)

Azure Container Apps does not currently support Azure Confidential Computing at the time of writing. If confidential container support is a requirement, Azure Container Instances (confidential SKU) or AKS with confidential node pools are the appropriate alternatives.

## Attestation

Before any TEE can receive a key or process sensitive data, it must prove it is genuine and unmodified — this is **remote attestation**. The TEE generates a hardware-signed report describing its configuration (CPU, firmware, loaded code). That report is sent to an attestation service, which validates it and returns a signed token.

Azure provides the **Microsoft Azure Attestation (MAA)** service for this. You can use shared regional providers (no setup required) or deploy your own for tighter control.

## Secure Key Release

Secure Key Release (SKR) is the mechanism that ties ACC and CMK together. It allows an HSM-backed key in Key Vault Premium or Managed HSM to be released — as plaintext, wrapped under a transport key — exclusively to an attested TEE.

The key has a **release policy** attached that specifies which attestation authority is trusted and what claims the TEE must satisfy. Key Vault will only release the key if both the attestation token is valid against the policy **and** the caller is an authorised identity.

The [next page](../skr) covers the specific release policy configuration for each TEE type.
